Your task is to extract evidence spans from this paper and classify each span into exactly
one of six categories. Use exactly these definitions — do not redefine them in your own words:

- "eval_metric": The definition of a measurement instrument, unit, or evaluation protocol (e.g.
  "top-1 accuracy", "pass@k accuracy", "FID score"). Naming/defining the instrument itself, not
  the number or result it produced.
- "pattern": The resulting trend, number, or direction that came out of a measurement (e.g.
  "gain of 6.2 points", "accuracy increases from X to Y", "monotonic increase", "FID dropped
  sharply after 10k steps"). What happened to the numbers, not the instrument used to get them.
  ("eval_metric" and "pattern" are jointly referred to as "observable" elsewhere in this
  pipeline, but "observable" is never a value you assign — always use "eval_metric" or "pattern".)
- "intervention": A specific action performed by the authors that is DIRECTLY paired with an
  isolated measured outcome — you must be able to point to one specific row/delta/figure-panel
  that resulted from performing exactly this action versus an alternative/baseline/prior
  condition (removed component, swapped module, different value, before/after). If you cannot
  name the specific comparison arm and the specific delta this action produced, it is NOT an
  intervention — even if it describes something the authors "did".
- "mechanism": A standing, always-on design principle or strategy that the authors' method
  embodies — stated abstractly enough that it could be reused to describe a DIFFERENT paper's
  approach, and NOT tied to one specific measured delta (its support instead comes indirectly,
  from multiple ablations elsewhere). Mechanism answers "what is our general approach / why
  should it work", never "what number did we get when we did X vs Y".
- "assumption": A premise about the problem/domain/data that some mechanism or intervention
  depends on, and that would still need to be true even if the authors had built a completely
  different method. If the sentence describes something the AUTHORS decided to do or build, it
  fails this test — reclassify as mechanism or context instead, even if originally phrased as a
  premise.
- "context": A concrete, paper-specific implementation/setup fact (exact dataset name, exact
  parameter count, exact token scheme, exact benchmark) that would need to be re-specified if the
  same mechanism were reused in another paper.

## R1 — Clause-level extraction (the unit is a ROLE, not a sentence)
- The extraction unit is the MINIMAL clause that performs exactly ONE of the six roles.
- Papers habitually pack several roles into one sentence. The most common pattern is:
  "we do X (intervention), and observe Y (pattern) on metric Z (eval_metric), suggesting
  W (mechanism)". You MUST split such sentences into separate spans — one span per role.
- Sibling spans coming from the same source sentence share a "parent" id. Parent ids use the
  source-sentence index (see "Per-category indexing" below), e.g. "S07". Standalone spans have
  "parent": null. Siblings must partition the sentence; their texts must not overlap.
- A span must never NEED a second label. If you are tempted to give one, split further.
- "eval_metric" and "pattern" are two separate categories, exactly like the assumption/context
  boundary — if a clause both names a metric and describes the trend it produced, you MUST split
  it into a separate eval_metric node and a separate pattern node, each with its own category id
  (E../P..). This split is mandatory, not optional (unlike in earlier versions of this task).
- Span text: verbatim quote strongly preferred, roughly 10–40 words. Paraphrase only for
  figures/tables that cannot be quoted, and append "(paraphrase)" to the text.

## R1b — Per-category indexing
- Every candidate sentence in the paper (or in this excerpt) is first assigned a sequential
  source-span index S1, S2, S3, ... in reading order (abstract -> body -> figures/tables ->
  appendix). This S-number is ONLY a source-span reference — it is never a node's own id.
- Each extracted node's real id is assigned per-category, sequentially in the order nodes of
  that category are produced:
  - assumption   -> A1, A2, A3, ...
  - context      -> C1, C2, C3, ...
  - mechanism    -> M1, M2, M3, ...
  - intervention -> I1, I2, I3, ...
  - eval_metric  -> E1, E2, E3, ...
  - pattern      -> P1, P2, P3, ...
- Every node records both ids: "node_id" (its category id, e.g. "A3") and "source_span" (the
  S-number it was extracted from, e.g. "S12"). Downstream stages reference nodes by "node_id"
  only.
- If one sentence S12 is split into two nodes of different categories (e.g. an intervention and
  an eval_metric), both nodes record "source_span": "S12", but each gets its own category id
  (e.g. "I4" and "E2").

## R1c — Experiment reference (experiment_ref)
Every intervention and pattern node MUST carry an "experiment_ref": a short stable id naming
the SINGLE experiment/ablation/comparison it belongs to. Nodes describing the SAME experiment
share the SAME experiment_ref, even when they appear in different sections (method vs. results),
figures, or pages.

- Derive it from the most specific anchor available, in this priority:
  table/figure number > named ablation > shared source sentence.
  Examples: "Fig7", "Table2", "TTT-offline-ablation", "patch-size-sweep".
- An intervention and the PATTERN(s) it produced MUST share one experiment_ref.
  Before finalizing, check: for each pattern, is there an intervention with the same
  experiment_ref? If the paper reports a result with no manipulation (pure observation:
  attention maps, t-SNE, qualitative examples), give it an experiment_ref that NO intervention
  shares — this correctly marks it as having no paired intervention.
- assumption / mechanism / context / eval_metric nodes do NOT get an experiment_ref (leave the
  field absent). eval_metric is a reusable measurement DEFINITION (e.g. "pass@1 accuracy"), not
  a fact owned by one experiment — the same metric legitimately appears across many experiments
  and papers, so it is never tied to a single experiment_ref.
- Do not invent a shared experiment_ref just to link nodes you WISH were one experiment. Same
  experiment_ref = you can point to the same table/figure/ablation for both.

## R1d — Direction preservation for comparative patterns
When a pattern node states a comparison, its text MUST name WHICH side wins / the direction of
the effect. Never reduce "A outperforms B by ~10 points" to "substantially better performance"
— the subject and direction are load-bearing and must survive both extraction and any later
generalization. A comparison pattern with the winner dropped is malformed; re-add the subject.

## R2 — Classification decision tree
Run every candidate clause through these steps, IN ORDER. Do not skip ahead. Do not force one
sentence into a single bucket, and do not duplicate the same clause into two categories.
Splitting a sentence into multiple nodes of different categories is expected and required
whenever it bundles more than one role.

STEP 1 — OBSERVABLE CHECK
Is this the definition of a measurement instrument/unit/protocol, OR the resulting trend/number/
direction from a measurement?
  Naming the instrument (e.g. "pass@k accuracy") -> "eval_metric" (category E).
  Describing the trend/result (e.g. "gain of 6.2 points", "accuracy increases from X to Y") ->
  "pattern" (category P).
  A clause that both names a metric AND describes its trend must be split into one eval_metric
  node and one pattern node — this split is mandatory (see R1), not a judgment call.
  NO to both -> go to STEP 2.

STEP 2 — INTERVENTION CHECK (contrastive-action test)
Is this a specific action performed by the authors that is DIRECTLY paired with an isolated
measured outcome — i.e., can you point to one specific row/delta/figure-panel that resulted
from performing exactly this action versus an alternative/baseline/prior condition (removed
component, swapped module, different value, before/after)?
  YES -> "intervention" (category I).
  NO  -> go to STEP 3.
  (Diagnostic: if you cannot name the specific comparison arm and the specific delta this action
  produced, it is NOT an intervention — even if it describes something the authors "did". If no
  intervention is reported for an otherwise clear claim, still note intervention: null on the
  surrounding factor — do not skip creating the surrounding nodes just because there is no
  intervention.)

STEP 3 — MECHANISM CHECK (standing abstract strategy test)
Is this a standing, always-on design principle or strategy that the authors' method embodies —
stated abstractly enough that it could be reused to describe a DIFFERENT paper's approach, and
NOT tied to one specific measured delta (its support instead comes indirectly, from multiple
ablations elsewhere)?
  YES -> "mechanism" (category M). If two competing explanations are given for the same result,
  create two separate M nodes. Before committing, run it through R2c below.
  NO  -> go to STEP 4.
  (Diagnostic: mechanism answers "what is our general approach / why should it work", never
  "what number did we get when we did X vs Y".)

STEP 4 — ASSUMPTION CHECK (choice-independent premise test)
Is this a premise about the problem/domain/data that some mechanism or intervention depends on,
and that would still need to be true even if the authors had built a completely different
method?
  YES -> "assumption" (category A). Hedging language ("may", "might", "could", "possibly") does
  NOT disqualify a clause from being an assumption — confidence in the phrasing is irrelevant;
  what matters is whether it survives the choice-independent test. Extract it either way and let
  "confidence"/"ambiguity_reason" (R3) carry your uncertainty about the classification itself,
  not the paper's own hedging. Before committing, run it through R2d below.
  NO  -> go to STEP 5.
  (Diagnostic: if the sentence describes something the AUTHORS decided to do or build, it fails
  this test — reclassify as mechanism or context instead, even if originally phrased as a
  premise.)

STEP 5 — CONTEXT CHECK (paper-specific instantiation test)
Is this a concrete, paper-specific implementation/setup fact (exact dataset name, exact
parameter count, exact token scheme, exact benchmark) that would need to be re-specified if the
same mechanism were reused in another paper?
  YES -> "context" (category C).
  NO  -> discard; not a node for any category.

CROSS-CHECK (STEP 3 vs STEP 5): if STEP 3 and STEP 5 both seem to fit the same sentence, split
it: the abstract strategy part -> mechanism; the concrete instantiation residue -> context.
Never let one node try to be both.

MANDATORY MIXED-SENTENCE RULE: many sentences (especially "Limitations" statements) bundle a
paper-specific setup fact with a domain premise/boundary claim, e.g. "we tested only on X, so
results may not hold on Y." Split such sentences at the clause boundary:
  - "we tested only on X" -> context (STEP 5), always extracted regardless of what happens to
    the other clause.
  - "results may not hold on Y" -> run through STEP 4. This IS an assumption (a choice-
    independent boundary condition about the domain/data) even though it is hedged with "may" —
    extract it as one, unless it is really just restating the authors' own scope choice, in which
    case it fails STEP 4 and stays out.

DUPLICATION RULE: never write the same full sentence into two different category fields as a
shortcut. If a sentence contains distinct clauses belonging to two categories, split it into
two nodes with two different (possibly partial) source texts — one goes to each category's
index — instead of copying the whole sentence into both.

CROSS-CHECK RULE (for consistency across nodes describing the same experiment/ablation): if two
or more sentences describe the same figure/table/ablation study (e.g. same Fig. reference), and
one sentence clearly states an intervention (STEP 2) for that experiment, all other nodes
referencing results from that same experiment must also be linked to that same intervention
index AND share its experiment_ref — do not leave intervention null for one sentence in the same
experimental unit while filling it in for another.

## R2b — "assumption" anti-patterns
- NEVER label as assumption: the paper title; contribution claims ("we show / we demonstrate
  that ..."); section headers; the authors' thesis or conclusions; anything describing what the
  authors decided to do or build. Those all fail the STEP 4 choice-independent test.
- Test before assigning: "Would this still need to be true even if the authors had built a
  completely different method?" Only domain/problem/data premises that pass this test qualify —
  including ones stated only in passing.

## R2c — "mechanism" anti-patterns (hard exclusions)
A mechanism is a STANDING design choice the method always embodies. The following are NOT
mechanisms even though they are interpretive prose — reclassify:

- An observed causal phenomenon or RISK that is a result/side-effect, not a design choice
  (e.g. "overtraining may cause the model to forget offline knowledge"). This describes what
  happens under a condition, not a strategy the method embodies.
  -> If it states a precondition some claim depends on, run STEP 4 (assumption); otherwise it is
     a diagnostic observation and usually should NOT be extracted as any of the six roles
     (discard), unless the trend itself is measurable (then it is a "pattern").

- A behavior observed in one specific figure/example (e.g. "the model interprets the rule as
  either going-through or touching" in Fig. X). This is a per-instance observation tied to one
  panel -> "pattern" (what was observed), NOT mechanism.

- A LEARNING GOAL or aspiration stated as what the method "attempts / tries to" do (e.g. "our
  method attempts to learn the relations between different tasks"). If this is something OBSERVED
  (e.g. shown in a t-SNE plot), it is a "pattern"; if it is just an aspiration with no standing
  mechanism behind it, discard. Do NOT file as mechanism.

- A comparative-superiority / positioning claim about the method vs. others (e.g. "our method
  does not rely on internet-scale data", "no recurrence is involved"). These describe
  positioning/contrast, not an always-on operating principle.
  -> If it is a concrete setup fact, STEP 5 (context); if it is a measured comparison outcome,
     STEP 1 (pattern); otherwise discard. Do NOT file as mechanism.

Test for mechanism (all three must hold): (1) the authors DECIDED to build it into the method,
(2) it is statable abstractly, (3) it is active across the whole method rather than observed in
one specific result. If any fails, it is not a mechanism.

## R2d — "assumption" must be the METHOD'S OWN premise
An assumption must be a premise THIS method relies on. A statement describing how a DIFFERENT /
competing approach works (e.g. "LLMs pretrained on internet-scale data learn transferable common
sense") is background about others, not a premise this method depends on -> classify as
"context", not "assumption", unless this method genuinely leans on that premise.

## R3 — Confidence and calibration
- Every span gets "confidence": your probability from 0.00 to 1.00 that assigned_label is
  correct.
- If confidence < 0.60, set assigned_label to "unresolved" — do not guess.
- Writing a substantive ambiguity_reason while reporting confidence >= 0.95 is a
  contradiction; recheck one of the two.

- alternative_labels_considered: only labels you genuinely weighed; usually an empty array.

- If the same sentence appears in more than one place (e.g., body and a figure caption),

- Only extract content that actually appears in this excerpt. Do not infer or reference content
