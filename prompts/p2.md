Below is a batch of Layer-1 AIO extraction nodes for one part of a paper: evidence nodes already
labeled with one category among assumption / mechanism / context / intervention / eval_metric /
pattern, each with a unique node_id (e.g. "A3", "M7", "C2", "I1", "E4", "P2") plus its exact
quoted "text" and "location". "eval_metric" and "pattern" are jointly called "observable"
elsewhere in this pipeline. node_ids are already globally unique across the whole paper — never
invent a new one and never renumber an existing one.

paper_id: {paper_id}
paper_title_hint: {paper_title_hint}

Perform BOTH tasks below over this same node batch, at once.

### TASK 1 — ASSEMBLE FACTORS BY SELECTING NODES (node-to-factor merging)

A factor is one single testable claim — roughly: one mechanism, tested under one intervention
(against one reference arm), in one context, with one observable result (an eval_metric and/or
the pattern it produced), resting on one assumption.

THE CORE RULE — SELECT, DO NOT SUMMARIZE, AND DO NOT REPRODUCE TEXT:
- "assumption", "mechanism", "context", and "intervention" must each be filled with the
  "node_id" of EXACTLY ONE node from the batch above — never its text.
- "observable" is an object with two sub-fields, "eval_metric" and "pattern"; each must be filled
  with the "node_id" of EXACTLY ONE node from the batch (an E.. id for eval_metric, a P.. id for
  pattern) — never its text.
- The full text for any node_id is already recoverable from the Layer-1 extraction files and must
  NOT be repeated here.
- NEVER merge several nodes of the same role into one field. NEVER paraphrase multiple nodes
  together. One field = one node's id.

MANDATORY COVERAGE RULE (verify before you finalize each factor):
- Every factor MUST have a non-null node id for "assumption", "mechanism", "context", and
  "intervention", AND at least one of "observable.eval_metric" / "observable.pattern" non-null.
- Do not fall back to null for one of these four roles just because no perfectly matching node
  sits right next to the rest of the group — first look again across the WHOLE batch (not just
  the immediate group) for the closest-fitting node for that role.
- If the batch truly contains no usable node for one of the four mandatory roles
  (assumption/mechanism/context/intervention) for a given experiment/claim, do not emit an
  incomplete factor for it — leave that group out of the factor list entirely instead.
- Before writing your final output, re-scan every factor you drafted against this rule and drop
  or fix any factor that still fails it.

HOW TO GROUP NODES INTO A FACTOR:
- Group nodes that describe the SAME single experiment / comparison / claim, then pick ONE node
  id per role (assumption / mechanism / context / intervention / observable.eval_metric /
  observable.pattern) from that group, subject to the MANDATORY COVERAGE RULE above.
- A mechanism node_id MAY appear in more than one factor — the same mechanism can legitimately
  explain results under different contexts/interventions/observables/assumptions. This is
  expected and correct; do not avoid reusing a mechanism id just because you already used it.
- NO DUPLICATE FACTORS: the combination (assumption, mechanism, context, intervention,
  observable.eval_metric, observable.pattern) of one factor must be different from every other
  factor's combination. Never output two factors with the exact same combination of node ids
  across all fields — if you find yourself about to do that, it is the same factor; merge them
  into one instead of repeating it.
- COMPETING OR DISTINCT MECHANISMS -> SEPARATE FACTORS. If a group of nodes contains two or more
  different mechanisms (different causal explanations) for the same experiment, output one factor
  PER mechanism; sibling factors may share the same context/intervention/observable/assumption
  node ids as long as each factor's overall combination stays unique (per the rule above).
- If two nodes state the SAME thing (mere restatement), use only one of them across all factors
  rather than creating duplicate factors — but see Task 2 below, which is where such duplicates
  should actually be resolved.
- One factor = one intervention axis x one condition. Do not fold two different comparisons into
  one factor just because they share a mechanism.
- CROSS-CHECK: if two or more nodes in the batch come from the same experimental unit (e.g. same
  figure/table, same source_span/parent) and one of them fixes an intervention node id, all
  factors built from that group must use that same intervention node id — do not leave
  intervention null for one factor in the group while filling it in for a sibling factor from the
  same experiment.
- Not every node needs to end up in a factor. If a node does not cleanly fit any experiment/claim
  grouping, simply leave it out rather than forcing a low-quality or single-node factor for it.

### TASK 2 — CANONICALIZATION + GENERALIZATION (run concurrently with Task 1, over the same node batch)

Some nodes in this batch may be near-duplicates: the same underlying claim expressed with
slightly different wording, or repeated more than once. Separately, almost every node's text was
quoted verbatim from THIS ONE paper, so it is more specific than it needs to be if the node is
ever reused when building a knowledge base spanning MANY papers.

MERGING DUPLICATES:
- There is no fixed similarity threshold — use your own judgment, given the paper's context, on
  whether two nodes denote the SAME underlying variable/claim/result (merge them) versus a merely
  related but distinct one (keep separate).
- For each duplicate cluster, pick ONE existing node's id as canonical — prefer the node whose
  text is the most complete/self-contained. Never invent a new node_id.

GENERALIZING TEXT (applies to the canonical node of every cluster, AND to any standalone node
whose text you judge would benefit from it):
- You MAY rewrite a node's text into a shorter, more general form: drop incidental,
  paper-specific detail that is not load-bearing for the claim itself (e.g. an exact dataset
  name, a specific numeric result, a model version number), so that the same node could
  plausibly be reused as-is when reading a different paper.
- Do NOT over-compress: keep almost all of the original meaning. If a detail is actually
  essential to what the claim asserts — the claim would become false, vague, or a materially
  different claim without it — keep the detail; this is the exception, and when in doubt, keep
  it rather than cut it.
- Never fabricate a fact that is not supported by the original text; never touch node_id.
- If you rewrote a node's text, record the new text as "canonical_text" in its node_registry
  entry (see below). If a node's text is left unchanged AND it has no duplicates, it does not
  need a node_registry entry at all.

- Record every cluster and/or rewrite in "node_registry": "canonical_node_id", the full
  "merged_node_ids" list (for a rewrite-only entry with no real duplicates, this is just
  [canonical_node_id] itself), "canonical_text" (include this key ONLY if you rewrote the text —
  omit it if the text is unchanged), and a short "reason".
- In Task 1's factor output, always use the canonical node_id for any node that belongs to a
  cluster — never a non-canonical member.

Output ONLY a JSON object, no prose before or after, in exactly this shape:

```json
{
  "factors": [
    {
      "factor_id": "F1",
      "assumption": "A3",
      "mechanism": "M7",
      "context": "C2",
      "intervention": "I5",
      "observable": {"eval_metric": "E4", "pattern": "P2"}
    }
  ],
  "node_registry": [
    {
      "canonical_node_id": "A3",
      "merged_node_ids": ["A3", "A9"],
      "canonical_text": "...",
      "reason": "..."
    }
  ]
}
```

JSON validity requirements: double quotes only, no trailing commas, escape internal quotes and
backslashes, no comments inside the JSON. Output nothing before or after this JSON object.
